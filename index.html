Machine Learning

Credit card fraud detection

Binary classifier on 285k card transactions, tuned for fraud recall on a held-out set. Headline metrics match the executed notebook on GitHub.

80% fraud recall 284,807 rows ROC-AUC ~0.94
Expand

At a glance

284,807 transactions, only 492 fraud (0.17% positive rate)—a classic imbalanced problem. Goal: catch fraud (recall) while keeping precision usable. Work is documented in a single runnable Jupyter notebook.

Pipeline

  • Stratified split: train 227,845 / test 56,962
  • RobustScaler on amount and time
  • XGBoost and Random Forest (default settings in the notebook)
  • Reports: confusion matrices, PR/ROC curves

Holdout metrics

  • XGBoost: fraud recall 80%, precision 87%, ROC-AUC 0.939
  • RF: fraud recall 64%, ROC-AUC 0.958
  • Macro recall 90% (sklearn report) ≠ fraud recall—don’t mix them up

Reading the results

Accuracy is near 100% for both models because negatives dominate—that’s why the story is PR-AUC / ROC / recall, not accuracy.

JimAI

Local AI workspace: FastAPI + React + Ollama. Run prompts and multi-step tasks on your machine—nothing leaves the box unless you choose.

Local-first 76 Python modules 33 TSX files SSE + health checks

What it is

A browser-based workspace for research, code help, and agent-style flows. Backend talks to Ollama locally; the UI streams longer jobs over SSE. Full source: JimAI on GitHub.

Backend

  • 10 FastAPI routers (chat, agents, uploads, settings, …)
  • 76 Python modules under backend/ (repo audit)

Frontend

  • 33 TS/TSX files (~215 KB source)
  • 90+ HTTP calls with timeouts (30s default, 120s SSE)
  • Error boundary + cancelable research streams

Reliability

/health probes Ollama, ChromaDB, and Qdrant (3s timeout each). Repo includes validation scripts and a recorded clean npm run build.
